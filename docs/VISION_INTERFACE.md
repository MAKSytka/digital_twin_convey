# Контракт машинного зрения

## Цель

Модуль зрения должен выдавать данные, достаточные для алгоритма управления, но не зависеть от Gazebo. Подготовленный ROS-тип — `BoxObservationArray`.

## Рекомендуемое соответствие JSON и ROS

Пример внешнего JSON:

```json
{
  "count": 1,
  "boxes": [
    {
      "id": 1,
      "center_cm": {"x": 120.0, "y": 32.0},
      "size_cm": {"width": 30.0, "height": 18.0},
      "angle_deg": -15.0
    }
  ]
}
```

Преобразование:

```text
center.x = center_cm.x / 100
center.y = center_cm.y / 100
length_m и width_m = размеры / 100
yaw_rad = angle_deg * pi / 180
```

## Что необходимо зафиксировать до интеграции

1. Положение начала координат изображения.
2. Направление роста X и Y.
3. Связь координат камеры с мировой системой Gazebo.
4. Какая сторона прямоугольника считается длиной.
5. Диапазон угла: `[-180, 180)`, `[-90, 90)` или другой.
6. Знак угла — по или против часовой стрелки.
7. Частота обновления.
8. Стабильность `id` между кадрами.
9. Поведение при частичном перекрытии товара.
10. Значение `confidence` и порог отбраковки.

## Рекомендуемая мировая система

После калибровки адаптер должен публиковать:

- X в метрах вдоль движения;
- Y в метрах поперёк матрицы;
- Z центра коробки;
- yaw в радианах против часовой стрелки вокруг `+Z`.

## Важное ограничение текущего типа

`BoxObservation` не содержит линейную и угловую скорость. Их можно оценивать по последовательным кадрам либо расширить интерфейс отдельным сообщением после согласования с алгоритмистом.

## Рекомендуемая архитектура

```mermaid
flowchart LR
    E[BoxDetector.exe или камера] --> J[JSON]
    J --> A[vision_adapter ROS node]
    A --> B[/singulator/boxes]
    B --> C[singulation_controller]
```

`BoxDetector.exe` и датасеты относятся к внешним бинарным артефактам. В GitHub должен находиться адаптер, пример JSON и описание версии бинарника.
