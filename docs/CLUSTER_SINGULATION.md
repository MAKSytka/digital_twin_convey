# Кластерный алгоритм сингуляризации

## Проблема текущей версии

Текущий контроллер строит одну глобальную очередь, сортируя активные товары прежде всего по координате `X`. Для поперечной волны, в которой несколько товаров имеют почти одинаковый `X`, такой подход не задаёт устойчивую цель для каждого товара на всём протяжении разделения.

После того как один товар немного вырывается вперёд, глобальный порядок может измениться. Кроме того, общая ячейка назначается ближайшему прогнозируемому товару. При плотном контакте это полезно как локальная эвристика, но не гарантирует, что вся волна получит согласованный набор продольных позиций до конца матрицы.

Наблюдаемое следствие: примерно 40% товаров в сложных сценариях выходят с недостаточным продольным разделением.

## Основная идея

Контроллер должен сначала преобразовать множество наблюдений в устойчивые продольные кластеры, а затем решать две связанные задачи:

1. разделять товары внутри каждого кластера;
2. сохранять безопасный интервал между соседними кластерами.

```text
BoxObservationArray
  -> построение кластеров по X
  -> фиксация порядка внутри кластера по Y
  -> расчёт целевых продольных слотов
  -> скорости товаров
  -> распределение скоростей по 56 ячейкам
```

## 1. Построение кластеров

Для каждого товара известны центр `x_i`, размеры и угол. Базовое правило пользователя:

```text
|x_i - x_j| < dx_cluster
```

означает принадлежность одной поперечной волне.

Практически лучше использовать два порога:

```text
cluster_dx_enter_m = 0.25
cluster_dx_exit_m  = 0.40
```

- новый товар добавляется в кластер при расстоянии по `X` меньше `dx_enter`;
- уже связанная пара остаётся в одном кластере до `dx_exit`;
- `dx_exit > dx_enter` создаёт гистерезис и предотвращает дрожание состава кластера.

Размеро-зависимый вариант учитывает продольные полупроекции:

```text
h_x = 0.5 * (|L cos(yaw)| + |W sin(yaw)|)
surface_dx = |x_i - x_j| - h_x_i - h_x_j
```

Тогда условие связи:

```text
surface_dx < cluster_surface_gap_m
```

Для первой реализации достаточно центра `X`, но размеро-зависимый критерий устойчивее для смеси маленьких и крупных коробок.

Кластеры строятся как компоненты связности. Если A близок к B, а B близок к C, все три товара относятся к одному кластеру.

## 2. Устойчивый идентификатор кластера

Кластер должен существовать между кадрами. Для него сохраняются:

```text
cluster_id
member_track_ids
rank_by_track_id
created_stamp
resolved_since
last_seen_stamp
```

Новый набор наблюдений сопоставляется со старым кластером по максимальному числу общих `track_id`.

Если к кластеру временно присоединился новый объект, существующие ранги не меняются. Новый объект вставляется по `Y` только между свободными соседними рангами либо получает последний свободный ранг. Полная перенумерация каждый кадр запрещена.

## 3. Нумерация внутри кластера

При создании кластера товары сортируются по убыванию `Y`:

```text
rank 1 -> максимальный Y
rank 2 -> следующий Y
...
rank n -> минимальный Y
```

При практически одинаковом `Y` используются стабильные критерии:

```text
(-Y, sequence, track_id)
```

Ранг определяет желаемый порядок выхода. Товар с рангом 1 должен стать первым по `X`, ранг 2 — вторым и так далее.

## 4. Целевые продольные слоты

Для каждого товара вычисляется продольная полупроекция `h_x`.

Для соседних рангов требуемое расстояние центров:

```text
d_target(k-1, k) = h_x(k-1) + h_x(k) + target_gap
```

Целевые позиции:

```text
x_target[1] = x_anchor
x_target[k] = x_target[k-1] - d_target(k-1, k)
```

`x_anchor` движется вперёд с номинальной скоростью транспортирования. В момент создания кластера он выбирается не позади самого переднего товара:

```text
x_anchor(t0) = max(x_i) + cluster_lead_offset_m
x_anchor(t)  = x_anchor(t0) + v_nominal * (t - t0)
```

Так алгоритм формирует конкретную геометрическую цель вместо простого локального ускорения и замедления.

## 5. Закон скорости внутри кластера

Для каждого товара:

```text
position_error = x_target - x
velocity_error = v_nominal - vx

v_command = v_nominal
          + cluster_position_gain * position_error
          + cluster_velocity_gain * velocity_error
```

Для ранга 1 можно добавить ограниченный лидерский boost. Команда обязательно ограничивается:

```text
positive_lag_floor <= v_command <= 3.0 м/с
```

Отрицательная скорость не используется.

Стартовая эвристика до ввода полного позиционного регулятора:

```text
rank 1: v_cluster + 0.5 * (n - 1) * speed_step
rank 2: v_cluster + 0.5 * (n - 3) * speed_step
...
rank n: v_cluster - 0.5 * (n - 1) * speed_step
```

После этого обратная связь по целевым слотам корректирует скорости.

Начальное значение:

```text
cluster_rank_speed_step_mps = 0.40
```

## 6. Ограничение продольного отставания

Существующий forward-only бюджет сохраняется. Каждый товар имеет виртуальную траекторию со скоростью `2 м/с` и допустимое отставание.

Кластерный регулятор не может назначить скорость ниже индивидуального `lag_limited_speed_floor`.

При конфликте требований приоритеты такие:

1. не допустить движения назад;
2. не допустить столкновения кластеров;
3. не превысить продольный бюджет отставания;
4. сформировать целевой внутрикластерный зазор;
5. исправлять yaw.

## 7. Межкластерная защита

Кластеры сортируются по передней границе:

```text
cluster_front_x = max(x_i + h_x_i)
cluster_rear_x  = min(x_i - h_x_i)
```

Для ведущего и следующего кластера:

```text
inter_cluster_gap = leading.rear_x - trailing.front_x
```

Если зазор меньше целевого, базовая скорость всего заднего кластера ограничивается. При этом она не опускается ниже максимального индивидуального lag-floor его участников.

Это предотвращает ситуацию, когда ускоренный товар из задней волны догоняет неразрешённую переднюю волну.

## 8. Разрешение конфликтов одной ячейки

Одна физическая ячейка не может одновременно выполнять две разные команды. Выбор владельца должен учитывать не только расстояние до центра ячейки, но и срочность:

```text
urgency = |x_target - x| / max(distance_to_exit, epsilon)
```

Приоритет предложения:

```text
1. товар из финализируемого кластера;
2. большая urgency;
3. больший overlap с ячейкой;
4. меньший rank;
5. меньшая дистанция до центра ячейки.
```

После выбора владельца команда всё равно проходит ограничитель ускорения `3 м/с²`.

## 9. Финализатор у выхода

Для каждого неразрешённого кластера вычисляется оставшаяся длина:

```text
distance_to_exit = matrix_max_x - cluster_front_x
```

При `distance_to_exit < cluster_finalizer_distance_m` контроллер:

- увеличивает позиционный gain;
- разрешает лидеру скорость до `3.0 м/с`;
- снижает задние ранги только до положительного lag-floor;
- временно уменьшает приоритет коррекции yaw;
- запрещает изменение рангов и состава, кроме исчезнувших треков.

Начальное значение:

```text
cluster_finalizer_distance_m = 1.20
```

## 10. Критерий разрешённого кластера

Кластер считается разрешённым, если для каждой соседней пары рангов:

```text
clearance >= cluster_target_gap_m
```

и условие сохраняется непрерывно:

```text
cluster_resolution_hold_s = 0.30
```

После разрешения кластер не формируется заново, пока товары не выйдут из зоны управления. Это предотвращает повторное объединение уже построенной цепочки.

## 11. Проверка достижимости

До назначения скоростей полезно оценивать, успеет ли кластер разделиться.

Требуемое дополнительное смещение:

```text
required_spread = sum(max(0, target_gap - current_clearance))
```

Оставшееся время:

```text
time_left = distance_to_exit / v_nominal
```

Минимальная требуемая разность скоростей:

```text
required_delta_v ~= required_spread / max(time_left, epsilon)
```

Если она больше доступного диапазона, контроллер должен зарегистрировать `cluster_infeasible` и запросить снижение входного потока либо увеличение дистанции между волнами. Нельзя скрывать физически недостижимый сценарий увеличением коэффициентов.

## 12. Предлагаемые параметры ROS

```text
enable_cluster_control            true
cluster_dx_enter_m                0.25
cluster_dx_exit_m                 0.40
cluster_target_gap_m              0.18
cluster_lead_offset_m             0.05
cluster_position_gain             2.0
cluster_velocity_gain             0.35
cluster_rank_speed_step_mps       0.40
cluster_resolution_hold_s         0.30
cluster_finalizer_distance_m      1.20
cluster_finalizer_gain_multiplier 1.6
cluster_max_size                  6
```

Значения являются начальными для симуляции и должны настраиваться по метрикам.

## 13. Диагностика

Контроллер должен публиковать или логировать:

```text
active_clusters
largest_cluster_size
resolved_clusters
unresolved_clusters_at_exit
cluster_resolution_ratio
minimum_intra_cluster_gap
minimum_inter_cluster_gap
cluster_infeasible_count
finalizer_active_clusters
```

Главная метрика:

```text
cluster_resolution_ratio = resolved_clusters / completed_clusters
```

Цель первого этапа — повысить долю сингуляризованных товаров с приблизительно 60% минимум до 90% на фиксированном наборе seed-сценариев.

## 14. План реализации в коде

В `singulation_controller.py` рекомендуется добавить:

```text
ClusterState
ClusterMember
_build_clusters(active_tracks)
_match_persistent_clusters(new_clusters)
_assign_cluster_ranks(cluster)
_update_cluster_targets(cluster, now_s)
_build_cluster_speed_commands(clusters)
_apply_inter_cluster_guard(clusters, speeds)
_is_cluster_resolved(cluster, now_s)
```

Первая версия должна быть включаемой параметром `enable_cluster_control`, чтобы можно было сравнивать старый и новый алгоритмы на одинаковых seed без изменения остальных компонентов симуляции.

## 15. План эксперимента

Для каждого значения `seed` выполнить два запуска:

```text
A: enable_cluster_control=false
B: enable_cluster_control=true
```

Фиксируются:

```text
число товаров
число разрешённых кластеров
число товаров без достаточного зазора на выходе
минимальный зазор
максимальное продольное отставание
число столкновений между волнами
средняя и максимальная команда скорости
```

Рекомендуемый набор:

```text
seed = 1, 2, 3, 4, 5, 42, 100
target rate = 1.0, 2.0, 3.0, 4.0 товара/с
```

Кластерный режим принимается как базовый только после устойчивого улучшения метрик без отрицательных скоростей и без роста межволновых столкновений.
