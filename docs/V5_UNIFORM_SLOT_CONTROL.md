# V5: единое управление всеми рядами и выходной конвейер

## Причина изменения

На записи интерфейса машинного зрения видно, что распознавание коробок работает устойчиво, однако на последних рядах матрицы регулярно образуется плотная группа перед роликовым горлышком. Особые правила V4 создавали два нежелательных эффекта:

- первые ряды выдавали общий повышенный разгон независимо от фактического расписания товара;
- последние ряды смешивали рассчитанную скорость со скоростью роликов и продолжали удерживать уже разделённые коробки по устаревшим временным слотам;
- постоянный ранговый профиль задавал скорость по номеру товара даже тогда, когда фактическая ошибка времени выхода требовала другого воздействия;
- повторное откладывание временных слотов задних кластеров могло накапливаться и приводить к ползущему движению у выхода.

## Принцип V5

Все 14 рядов используют один и тот же закон управления. Номера рядов не влияют на скорость.

```text
BoxObservationArray
  -> устойчивый кластер и фиксированный порядок по Y
  -> назначение временных слотов выхода
  -> required_speed = distance_to_exit / time_to_slot
  -> ограничение 1.0..3.0 м/с
  -> распределение команды по ячейкам под коробкой
```

Удалены:

```text
capture_speed_mps
capture_rows
stabilisation_rows
rank_feedforward_weight
stabilisation_output_weight
stabilisation_yaw_multiplier
накопительное смещение slot_time
```

Сохранены:

```text
minimum_speed_mps         = 1.0
maximum_speed_mps         = 3.0
maximum_acceleration_mps2 = 6.0
MAX_JERK                  = 30.0
```

## Выпуск разрешённого кластера

После того как все соседние товары внутри кластера удерживают заданный зазор, кластер помечается `resolved`. В V4 он продолжал следовать старым временным слотам и мог замедляться около выхода.

В V5 разрешённый кластер получает общую скорость:

```text
discharge_speed_mps = 2.5
```

Это не отдельная зона матрицы: правило зависит от состояния кластера и одинаково действует на любом ряду.

## Межкластерная защита

Накопительное откладывание расписания удалено. Защита теперь включается только при фактическом нарушении жёсткого межкластерного зазора:

```text
inter_cluster_hard_gap_m = 0.12
```

Тогда скорость заднего кластера временно ограничивается относительно скорости переднего. Когда расстояние восстановлено, ограничение снимается.

## Выходной конвейер

После роликов добавлен обычный конвейер:

```text
длина   = 1.60 м
ширина  = 0.60 м
скорость по умолчанию = 2.50 м/с
топик   = /singulator/outfeed/cmd_vel
```

Он нужен для наблюдения, сохраняются ли зазоры после роликов и не возникают ли повторные столкновения.

Перенос GUI-камеры:

```bash
./scripts/view_outfeed.sh
```

Верхняя камера машинного зрения не расширялась: изменение её поля могло бы нарушить уже отлаженную калибровку. Новый участок наблюдается через Gazebo GUI.

## Испытания

Сначала собрать проект и выполнить:

```bash
TARGET_RATE_BOXES_PER_SEC=2.0 SEED=42 ./scripts/run_roller_demo.sh
TARGET_RATE_BOXES_PER_SEC=3.0 SEED=42 ./scripts/run_roller_demo.sh
TARGET_RATE_BOXES_PER_SEC=4.0 SEED=42 ./scripts/run_roller_demo.sh
```

Во втором терминале:

```bash
./scripts/check_v5_control.sh
./scripts/view_outfeed.sh
```

В логах контроллера используется префикс `control_v5`.

Для каждого запуска фиксировать:

```text
seed
интенсивность
unresolved
slot_error
overloaded
intra_gap
inter_gap
conflict_cells
число пар коробок на выходном конвейере без требуемого зазора
```

## Критерий следующего решения

Если сгущение исчезнет, но отдельные кластеры всё ещё не разделяются, следующий шаг — корректировка механизма назначения слотов и владения общей ячейкой. Возвращать специальные первые или последние ряды не следует без измеримого преимущества на одинаковых seed.
